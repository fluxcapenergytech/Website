# FluxTech Website — flux-tech.de

Single-page marketing site for FluxTech, hosted on GitHub Pages. Originally exported from Carrd, now maintained via a content pipeline.

## Architecture

```
flux-tech.de/
├── CLAUDE.md              # You are here
├── content.yaml           # Source of truth for all site text
├── build.py               # Assembles sections → injects YAML content → writes index.html
├── sections/              # Section-based HTML partials (replaces monolithic template.html)
│   ├── head.html          # Doctype, <head>, opening body/wrapper tags, inline CSS, SVG defs
│   ├── home.html          # home-section
│   ├── problem.html       # problem-section
│   ├── solution.html      # solution-section
│   ├── businessmodel.html # businessmodel-section
│   ├── physicsengineering.html # physicsengineering-section
│   ├── market.html        # market-section
│   ├── team.html          # team-section
│   ├── ask.html           # ask-section (final slide/CTA + credits)
│   └── closing.html       # Closing wrapper tags, all inline <script> blocks
├── index.html             # Built output — served by GitHub Pages. NEVER edit directly
└── assets/
    └── images/            # Logo, headshots, IEA chart, partner logos, social card
```

## Content Pipeline

`content.yaml` is the single source of truth for all visible text. To change site copy:

1. Edit `content.yaml`
2. Run `python build.py`
3. Commit and push — GitHub Pages serves the updated `index.html`

`sections/*.html` are the Carrd-exported HTML partials with text replaced by `{{key}}` placeholders. They contain all CSS, JS, layout, and image references. `build.py` concatenates them in a defined order (`SECTION_ORDER` list in `build.py`) to form the full template, then performs placeholder replacement from `content.yaml`.

- To edit a section's structure or styling, edit `sections/<name>.html`.
- To edit text content, edit `content.yaml` (unchanged).
- CDN script tags can be added to any section partial or to `sections/head.html`.
- Section order is defined by the `SECTION_ORDER` list in `build.py`.

`index.html` is a build artifact. Every line is generated by `build.py`. Manual edits will be overwritten on next build.

## YAML Content Format

Content uses markdown-style inline formatting, converted to HTML by `build.py`:

| Markdown | HTML output |
|---|---|
| `**bold text**` | `<strong>bold text</strong>` |
| `==highlighted text==` | `<mark>highlighted text</mark>` |
| `[link text](url)` | `<a href="url">link text</a>` |
| `[link text](mailto:x)` | `<a href="mailto:x">link text</a>` |

Raw HTML (e.g., `<br />`, `<sub>`, `<sup>`) is allowed in YAML values when no markdown equivalent exists. The build script passes these through unchanged.

Paragraphs within a single YAML value are separated by `\n\n`. The build script wraps each paragraph in `<span class="p">...</span>` — this is required by the Carrd CSS.

## HTML Structure

The site has sections split into individual partials under `sections/`: home, problem, solution, businessmodel, physicsengineering, market, team, and ask.

Text elements use two ID patterns:
- **Numbered IDs** (`text15`, `text26`, `text07`): Carrd auto-generated. These map to generic structural elements — tagline, body paragraphs, captions.
- **Semantic IDs** (`text-bm-customer`, `text-bm-fluxtech`, `text-problem`): Human-assigned descriptive IDs for the business model, problem, solution, and other content sections.

The YAML keys match the element IDs. Example:

```yaml
text15: "Converting AI data centre waste heat into electricity."

text-bm-customer: |
  **For the Customer.**

  No capital outlay. No operational burden...
```

## Build Script Behavior

`build.py` assembles the full template by concatenating `sections/*.html` partials in `SECTION_ORDER`, processes markdown formatting, wraps paragraphs in `<span class="p">` tags, and replaces `{{element_id}}` tokens to produce `index.html`.

The script validates that every YAML key has a matching placeholder in the assembled template and errors on orphaned placeholders.

## Commands

```bash
python build.py              # Build index.html from sections + content
python build.py --check      # Dry run: validate YAML keys match template placeholders
```

## Conventions

- Fonts: Poppins (headings, gradient text) and Source Code Pro (body).
- Brand palette: teal-to-green gradient (`#007FAD` → `#009A62`), body text `#1A1D21`, highlight background `#CBF5E6`.
- Section headings use Poppins with CSS gradient fill. Body text uses Source Code Pro.
- All images live in `assets/images/` with descriptive kebab-case names (e.g., `logo-fluxtech.jpg`, `headshot-john.jpg`, `chart-iea-datacenter-energy.jpg`). No auto-generated numbered filenames.
- The YouTube embed URL lives in `sections/home.html` (it's structural, not copy).

## What to Preserve

The Carrd CSS depends on specific HTML structure. In particular:

- Text paragraphs use `<span class="p">` inside `<p>` or `<h2>` tags. Switching to `<p>` tags breaks spacing.
- `<mark>` tags trigger the `#CBF5E6` highlight background via CSS class `.instance-29 mark` (and similar).
- Gradient headings use `-webkit-background-clip: text` — they require `<span class="p">` wrapping to render correctly across paragraph breaks.
- Element IDs (`id="text-bm-customer"` etc.) are referenced by CSS instance classes. Changing an ID requires updating the corresponding CSS rule.
